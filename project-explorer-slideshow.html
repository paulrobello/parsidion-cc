The Problem: Pattern Reinvention
When working across multiple projects, useful implementation patterns are reinvented from scratch because there's no record of how a prior project solved the same problem. Each session starts with zero cross-project context.
Common scenarios
  • Built a layered config system in par-llama — re-invented it 3 months later in another project
  • Wrote a Rich progress CLI — forgot the exact pattern by the next project
  • Solved PTY spoofing in parvault — nothing to reference when it came up again
The fix
project-explorer → "explore this project and save it to the vault" Step 1: vault check (no duplicate work) Steps 2-5: deep project analysis Steps 6-7: write overview + pattern notes Step 8: rebuild index (notes are searchable) Future: vault-search "streaming output" → finds the note

Agent: 9-Step Workflow
1Vault Check — semantic search + dispatch vault-explorer; fill gaps only
2Metadata Discovery — README, pyproject.toml, Cargo.toml, Makefile
3Architecture Exploration — Glob + Read, top-level dirs, key modules
4Feature Extraction — 3–8 reusable features, binary --help, Makefile targets
5Pattern Identification — config, error handling, logging, design patterns
6Write Overview Note → Projects/{slug}-overview.md
7Write Pattern Notes → Patterns/{feature-slug}.md (min 3)
8Rebuild Index — update_index.py
9Summary Report — paths created/updated, skipped features, index status
Steps 1–2: Vault Check + Metadata
Step 1 — vault check before writing anything
uv run --no-project ~/.claude/skills/parsidion-cc/scripts/vault_search.py \ "project {name}" -j 2>/dev/null # Also dispatch vault-explorer: "project {name} architecture features" # If notes exist → read, identify gaps, only add/update (never delete) # If no notes → proceed to full analysis
Step 2 — files read for metadata extraction
README.md / README.rst → description, features, usage CLAUDE.md → conventions, architecture notes pyproject.toml → name, version, deps, entry points Cargo.toml / package.json / go.mod Makefile → available commands Extract: project-slug · language+version · frameworks key deps (top 5-10) · entry points · build tool
Steps 3–4: Architecture + Features
Step 3 — architecture exploration
# 1. List top-level dirs (one level deep) # 2. Read one representative file per significant dir # 3. Identify main entry point(s) # 4. Note distinctive structural choices: # monorepo · plugin system · layered config · event bus
Step 4 — feature extraction sources
README feature sections {binary} --help 2>/dev/null || true Key module filenames + light content skim Makefile targets Per feature: what it does · which files · why reusable
3–8 features is the target range — enough to be useful, not so many that the vault fills with noise. Features that are too project-specific are skipped and reported in the summary.
Step 5: Pattern Identification
Beyond individual features, the agent looks for project-level patterns — recurring solutions that appear across many projects and are worth documenting as reusable building blocks.
Pattern categories
◆Config handling — YAML, env vars, layered precedence, --config flag
◆Error handling — exceptions, Result types, exit codes, structured errors
◆Logging — Rich, tracing, structured JSON, file-based, debug flags
◆Design patterns — plugin arch, event-driven, strategy, middleware chain
◆Testing approach — unit vs integration, fixtures, mocking conventions
◆CLI conventions — argument parsing, subcommands, output formatting
Patterns identified here drive which Patterns/ notes get written in step 7.

Quality Rules
The agent enforces five hard rules to keep the vault healthy. These prevent the most common failure modes found in vault notes: orphan notes, empty sections, and tilde paths that break downstream tools.
1No orphan notes — every related field must contain at least one [[wikilink]]
2No empty sections — omit a heading rather than leaving it blank
3Absolute source paths — sources must use full filesystem path, not ~
4Kebab-case filenames — no date suffix (date goes in frontmatter)
5Search before create — update existing pattern notes; add related link back to overview
Related field format — inline quoted array (per vault convention): related: ["[[note-a]]", "[[note-b]]"] ✓ related: [[[note-a]], [[note-b]]] ✗ (bare) related: ✗ (empty)

Key design decisions
  • Uses Sonnet (not Haiku) — deep reasoning needed for architecture analysis
  • vault-first — checks for existing notes before writing anything; updates, not duplicates

  • Bidirectional links — overview ↔ pattern notes via wikilinks
  • update_index.py always last — notes immediately searchable via vault-search
